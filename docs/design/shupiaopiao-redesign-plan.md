# 书漂漂小程序设计与交互改造方案

更新时间：2026-06-28

## 1. 改造目标

本方案用于指导后续 UI、交互和轻量业务逻辑调整。当前阶段仅作为设计方案文档，不代表已执行代码修改。

核心目标：

- 让新用户在首屏快速理解：闲置书可以上漂，接漂需要公益积分，运费采用快递到付。
- 让老用户更快完成两条主路径：接漂找书、从书架上漂赠书。
- 保持现有合规边界：不使用购买、余额、交易、提现等价格化表达；不新增读书笔记、公开社交流、用户自定义图片上传等敏感能力。
- 保持现有产品术语：公益积分、信用积分、流转积分、占用中、快递到付。

## 2. 本版修正原则

相对于前序方案，本版固定以下三点：

1. 漂流页的一句价值主张和“玩法介绍”入口与“漂流广场”并排一行，通过字号和按钮样式区分。
2. 漂流广场的可用积分和“获得积分”入口与当前版本保持一致，不重构成新的积分卡片。
3. 书架页面仅调整两层筛选的默认设计：一级筛选为“推荐 / 具体的书籍分类 / 更多筛选”。除此之外，其他书架结构和能力保持前序建议。

除以上三点外，其他页面建议沿用前序评估方案。

## 3. 全局设计原则

### 3.1 信息层级

- 每个页面最多突出一个主行动按钮。
- 主按钮只用于推进核心流程：申请接漂、确认申请、提交上漂、去发货。
- 次按钮用于低承诺行为：想要、查看规则、稍后处理、查看详情。
- 绿色只用于主行动和公益积分相关信息，不用于所有标签。
- 警示信息使用浅黄或浅橙，避免和公益积分的薄荷绿混用。

### 3.2 文案原则

- 避免交易化表达：不使用“购买”“价格”“付款给平台”“余额”等词。
- 运费说明保持明确但克制：可说明“快递到付”“收件时向快递员支付”，不展示具体金额估算。
- 公益积分说明保持站内流转语义：接漂先占用，完成后记录结算。
- 信用积分说明保持履约语义：发货、收货、申诉、取消等行为会影响履约记录。

### 3.3 导航与任务优先级

- 首页优先承接“找书接漂”。
- 书架优先承接“管理藏书”和“从已有书发起上漂”。
- 我的页优先承接待办，包括待发货、待确认、待评价、申诉等。

## 4. 漂流广场

对应页面：`miniprogram/pages/pool/index`

### 4.1 当前问题

- 首屏信息密度较高，但目前“漂流广场 / 玩法介绍 / 可用积分 / 获得积分”的结构已经形成稳定布局。
- 两层筛选逻辑相对复杂，新用户需要先理解“按品类 / 按价值 / 按品相”才能筛书。
- 书卡主行动已明确为“申请接漂”，这是合理的主转化入口，应保留。

### 4.2 改造方案

首屏结构建议：

```text
漂流广场   闲置书送出去，好书漂回来   [玩法介绍]
剩余可用公益积分 12                 [获得积分]

搜索书名 · 作者 · ISBN

推荐 / 童书 / 文学 / 社科 / 更多筛选

双列图书卡片
书封
书名
流转积分 + 分类
[申请接漂]

悬浮入口：[＋ 上漂赠书]
```

### 4.3 重点修正

- “漂流广场”、一句价值主张、“玩法介绍”在同一行。
- “漂流广场”继续作为大标题。
- 价值主张使用更小字号，建议 22rpx 左右，颜色使用中性灰绿色，单行省略。
- “玩法介绍”继续保持按钮/胶囊样式，与文字主张形成视觉区分。
- 可用积分和“获得积分”入口保持当前结构。
- 可用公益积分和“获得积分”入口保持当前结构：一行展示，不改为积分卡片。

### 4.4 筛选建议

- 默认露出高频入口：推荐、童书、文学、社科、更多筛选。
- “更多筛选”承载价值区间和品相筛选。
- 列表主按钮继续使用“申请接漂”，不在列表卡片上使用“想要接漂”作为主入口。

### 4.5 空状态文案

```text
漂流池暂时没有符合条件的书
```

按钮：

```text
去上漂一本书
```

## 5. 漂流详情

对应页面：`miniprogram/pages/pool/detail`

### 5.1 当前问题

- “想要接漂”和“申请接漂”并列时，用户可能误判两者是同等优先级。
- 积分占用、运费到付、寄出地、品相、赠书方信用等信息分散，需要用户自行理解规则。
- “举报”入口需要保留，但不应抢占核心决策区域。

### 5.2 改造方案

页面结构建议：

```text
大书封

书名
作者
流转积分 6   品相良好   轻量
从 上海 寄出 · 快递到付

接漂前知道
- 申请成功后会先占用 6 公益积分
- 运费为快递到付，收件时向快递员支付
- 确认收货后完成本次公益积分记录

内容介绍
基础信息
同一赠书方还有这些书
赠书方信息

底部：[想要] [申请接漂]
```

### 5.3 交互建议

- 底部主按钮“申请接漂”占更大宽度。
- “想要”作为弱按钮或图标按钮，表达收藏意愿，不干扰主转化。
- 如果公益积分不足，主按钮可降级为“积分不足，先加入想要”，并引导“获得积分”。
- 举报入口放在详情页底部弱链接区域，保持合规但降低视觉权重。

### 5.4 关键文案

规则卡标题：

```text
接漂前知道
```

规则文案：

```text
申请成功后会先占用 {{coinValue}} 公益积分
运费为快递到付，收件时向快递员支付
确认收货后完成本次公益积分记录
```

同赠书方提示：

```text
同一赠书方还有这些书，可一起申请，通常只需一次到付运费
```

## 6. 申请接漂

对应页面：`miniprogram/pages/drift/claim`

### 6.1 当前问题

- 地址、运费、积分占用分散在多个区域，确认前缺少总览。
- “确认申请”之前，用户不容易快速判断自己将承担什么。
- 微信地址读取和地址管理入口相对分裂。

### 6.2 改造方案

页面结构建议：

```text
书籍摘要
书封 / 书名 / 流转积分 / 轻量标识

收货地址
姓名 手机
地区 详细地址
[更换地址] [读取微信地址]

本次申请摘要
将占用公益积分：6

寄出参考地：上海
同时未收货：最多 5 单

到付说明
平台不收取物流费用，赠书方寄出到付件，收件时向快递员支付运费。

[确认申请接漂]
```

### 6.3 交互建议

- 地址卡内合并“选择地址”和“读取微信地址”。
- 没有地址时，确认按钮置灰或点击后明确提示“先选择收货地址”。
- 公益积分不足时，不让用户走到失败反馈，提前展示“可用公益积分不足，去获得积分”。
- 申请成功后跳转到“我接到的书”，并展示申请成功提示。

### 6.4 关键文案

地址空状态：

```text
请选择收货地址，用于赠书方寄出到付件
```

摘要标题：

```text
本次申请摘要
```

按钮：

```text
确认申请接漂
```

成功提示：

```text
申请已提交，等待赠书方发货
```

合并提示：


## 7. 上漂赠书

对应页面：`miniprogram/pages/drift/publish`

### 7.1 当前问题

- 当前表单较长，用户不容易判断完成上漂需要几步。
- 批量上漂能力有价值，但首次上漂时容易提高理解成本。
- 定价缺失、补录信息、系统建议积分之间的关系需要更清晰。

### 7.2 改造方案

页面结构建议：

```text
步骤：选书 → 品相 → 积分 → 确认

1 选择要送出的书
书籍列表 / 已选状态 / 可继续选择多本

2 描述品相
几乎全新 / 良好 / 明显使用痕迹
可选：有笔记 / 有折角 / 有脏污 / 有破损

3 公益积分
系统按图书定价和品相给出建议积分
本次设定：[-] 6 [+]

4 漂流设置
匿名漂流 / 昵称漂流
寄出参考地

[提交上漂]
```

### 7.3 交互建议

- 首次进入时默认强调单本上漂，多选保留为增强能力。
- 定价缺失应在选择阶段明确阻断或强提示，不建议提交时静默自动取消。
- 仅在用户将流转积分设为 0 时增加确认弹窗，避免普通流程过重。
- 匿名漂流继续作为默认选项，减少用户心理负担。

### 7.4 关键文案

页面标题：

```text
上漂赠书
```

选书标题：

```text
选择要送出的书
```

信息缺失提示：

```text
缺少定价，补录后才能上漂
```

积分说明：

```text
系统按图书定价和品相给出建议积分
```

匿名说明：

```text
对外显示为匿名书友，不影响发货信息
```

## 8. 我的书架

对应页面：`miniprogram/pages/shelf/index`

### 8.1 当前问题

- 书架偏“藏书管理”，和“从已有书发起上漂”的连接还可以更强。
- 当前两层筛选以管理维度为主，默认筛选对普通用户不够直觉。
- 添加入口合理保留，但已有书转上漂的路径可继续加强。

### 8.2 本版筛选修正

书架页面仅调整两层筛选默认设计：

一级筛选建议：

```text
推荐 / 童书 / 文学 / 社科 / 经管 / 科普 / 更多筛选
```

其中：

- “推荐”默认展示全部或按后续推荐规则排序后的藏书。
- “具体的书籍分类”直接作为一级入口，减少用户先选择“书籍分类”再选择分类的步骤。
- “更多筛选”承载原有阅读状态和书架位置筛选。

二级筛选建议：

```text
当选择“推荐”：全部
当选择具体书籍分类：全部 / 可上漂 / 已上漂（可作为后续增强）
当选择“更多筛选”：阅读状态 / 书架位置
```

如果后续实现需要保持较小改动，也可以先采用：

```text
一级：推荐 / 书籍分类 / 更多筛选
二级：
- 推荐：全部
- 书籍分类：全部 / 童书 / 文学 / 社科 / 经管 / 科普 / 生活 / 艺术 / 其他
- 更多筛选：阅读状态 + 书架位置
```

### 8.3 书架卡片建议

书卡结构建议：

```text
书封
书名
作者 · 阅读状态
[上漂这本] / [查看漂流]
```

### 8.4 交互建议

- 书卡轻触仍进入书籍详情。
- 卡片操作区可直达“上漂这本”。
- 已上漂书籍的操作切换为“查看漂流”。
- 如果容量不足，添加入口先提示“容量不足，可用公益积分兑换额度”。

### 8.5 空状态文案

无可上漂书：

```text
这里还没有可上漂的书
```

按钮：

```text
去添加书籍
```

## 11. 公益积分明细

对应页面：`miniprogram/pages/mine/wallet`

### 11.1 当前问题

- “总记录值 / 占用中 / 可用公益积分”的关系对普通用户不够直观。
- 明细列表直接展示流水，对规则解释帮助有限。

### 11.2 改造方案

页面结构建议：

```text
可用公益积分
12
占用中 6 · 累计记录 38
[获得积分]

公益积分怎么用
- 接漂时先占用
- 收货完成后记录扣除
- 上漂完成后记录增加

明细
完成赠书记录 +6
接漂占用
取消接漂释放占用
```

### 11.3 文案建议

- “总记录值”建议改为“累计记录”。
- “余额”类文案继续避免使用。
- 明细描述统一为公益积分语义。

建议流水类型文案：

```text
完成赠书记录
接漂占用
取消接漂释放占用
确认收货记录扣除
申诉关闭释放占用
```

### 11.4 后续增强

可增加明细筛选：

```text
全部 / 增加 / 占用 / 释放 / 扣除
```

该项可作为 P2，不影响首批改造。

## 12. 非 UI / 交互层面的逻辑改动

### 12.1 漂流广场

- 如新增“推荐”筛选，需要定义推荐排序策略。
- 推荐可考虑：可接漂优先、有封面优先、轻量书优先、同城或近距离优先、赠书方信用良好优先。
- “更多筛选”需要承载原有价值区间和品相筛选，避免能力丢失。

### 12.2 申请接漂

- 申请页需要聚合地址、可用公益积分、占用积分、寄出地、未收货上限。
- 确认按钮状态应依赖：是否选择地址、可用公益积分是否足够、是否达到未收货上限。
- 公益积分不足时，应提前引导“获得积分”或“加入想要”，减少提交后失败。

### 12.3 上漂发布

- 发布页需要维护步骤状态：选书、品相、积分、确认。
- 定价缺失应前置阻断或明确提示，不建议提交时静默自动取消。
- 从书架卡片进入上漂时，需要携带 `shelfBookId` 并预选对应书籍。

### 12.4 书架

- 如果采用“推荐 / 具体分类 / 更多筛选”，需要调整一级筛选与二级筛选的数据结构。
- “更多筛选”需要保留原有阅读状态和书架位置能力。
- 如新增“可上漂 / 已上漂”筛选，需要基于 `canPublishDrift` 和漂流状态计算。

### 12.5 我的与订单

- 我的页待办模块需要按状态聚合，并支持点击直达订单列表对应 tab。
- 订单卡需要增加或前端计算 `nextStep` 文案。
- 合并发货卡需要展示 `orderCount`、同一收件地址、合并发货入口。

### 12.6 公益积分

- “累计记录”仍可沿用现有 `balance` 数据，只调整展示文案。
- 明细筛选如落地，需要按交易类型归类：增加、占用、释放、扣除。
- 所有积分流水描述继续避免钱包、余额、购买式表达。

## 13. 建议落地顺序

### V1：首批体验优化

- 漂流广场头部信息层级微调。
- 漂流详情底部按钮主次优化。

## 14. 验收标准

### 14.1 体验验收

- 新用户进入漂流广场后，能在首屏看到“漂流广场”、价值主张、玩法介绍、可用积分和获得积分入口。
- 用户在漂流详情页能明确知道：申请会占用公益积分，运费为快递到付，完成后再结算积分记录。
- 用户在申请页确认前能看到本次申请摘要。
- 用户在书架页能更快按推荐或具体分类浏览藏书，并能在更多筛选中找到阅读状态和书架位置。
- 用户在我的页能优先看到待处理事项。

### 14.2 合规验收

- 不出现“购买”“提现”“余额”“付款给平台”等交易化表达。
- 不新增公开笔记、公开评论流、用户图片社交分享能力。
- 运费说明不展示具体金额估算。
- 公益积分仍定义为站内流转记录，不具备现金属性。

### 14.3 技术验收

- 页面路由不变。
- 现有接漂、上漂、发货、确认收货流程不被破坏。
- 原有合同测试需按新产品约束更新：尤其是漂流首页头部布局和书架筛选默认项。
- 若涉及云函数数据字段变化，应补充对应合同测试；仅前端展示调整则优先保持接口不变。
