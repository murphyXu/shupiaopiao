# 接漂链路文案精简 · 设计说明 v1

> 范围：`pool/detail`（漂流详情）→ `drift/claim`（申请接漂）→ 确认弹窗（`buildClaimConfirmContent`）  
> 目标：递进披露、术语统一、去重复，不削弱合规与关键决策信息。

---

## 1. 设计原则（ recap ）

| 原则 | 说明 |
|------|------|
| **A. 递进披露** | 详情页讲「是否值得接」；申请页讲「填什么、占多少」；弹窗只做「最后确认」 |
| **B. 术语统一** | 到付、占用、记录、扣除 — 全链路同一套说法，集中维护 |
| **C. 每类信息只出现一次** | 同一规则不在相邻步骤重复 2 遍以上 |
| **D. 合规不变** | 仍用「申请/占用/记录」，不用「领取/扣款/余额/需 X 积分」 |

## 2. 统一术语表（`driftCopy.js` 草案）

| Key | 文案模板 | 使用场景 |
|-----|----------|----------|
| `COD_SHORT` | 快递到付，收件时付运费 | 详情 meta、申请摘要「运费方式」 |
| `COD_PLATFORM` | 平台不代收运费 | 详情 know-card 第 2 条 |
| `COIN_OCCUPY` | 先占用 **{n}** 公益积分 | know-card、摘要、弹窗 |
| `COIN_SETTLE` | 确认收货后从占用中结算 | 弹窗第 2 行 |
| `INFLIGHT_LIMIT` | 在途接漂最多 5 单 | know-card、摘要（仅一处） |
| `SAME_GIVER_HINT` | 分别申请接漂，地址相同可合并一次寄出 | 同赠书方卡片默认 hint |
| `ADDRESS_PLACEHOLDER` | 选择收货地址 › | 申请页地址空态 |
| `SHIPPING_SAME_CITY` | 与您同城寄递，到付运费通常较低 | `shippingDistanceHint` 同城 |
| `SHIPPING_CROSS_PROVINCE` | 跨省寄递，到付运费通常高于同城 | `shippingDistanceHint` 跨省 |
| `SHIPPING_FOOTNOTE` | 实际运费以快递员收费为准 | hint 后缀（可选，与 hint 合并时加在句末） |
| `CLAIM_MODAL_CONFIRM` | 是否确认申请？ | 弹窗末行 |

**收货后结算**（弹窗专用，与「占用」配对）：

- 页面侧统一说「占用 → 完成记录」
- 弹窗使用：`确认收货后从占用中结算`（不用「扣除」语义）

## 3. 页面设计稿

### 3.1 漂流详情 `pool/detail`

#### 结构（变更标注）

```
┌─────────────────────────────┐
│ [封面]                       │
├─────────────────────────────┤
│ 书名 / 作者                  │
│ 公益积分 12 · 九成新 · 轻量   │
│ 从 杭州 · 西湖区 寄出 · 到付  │  ← 保留，COD_SHORT 化
│ 赠出内容：…（如有）           │
├─────────────────────────────┤
│ ▌接漂须知                    │  ← 原「接漂前知道」，标题略短
│ • 先占用 12 公益积分，收货后   │
│   完成积分记录               │  ← 合并原 know 第 1、3 条
│ • 快递到付，平台不代收运费     │  ← 合并原 know 第 2 + tip 核心
│ • 在途接漂最多 5 单           │  ← 从申请页前移到此处（决策信息）
├─────────────────────────────┤
│ 内容介绍 …                   │
│ 基础信息 …                   │
│ 同赠书方其他书（如有）        │
│ 赠书方信息                   │
├─────────────────────────────┤
│ [了解更多漂流规则 ›]          │  ← 新增：链到 guide，替代长 tip
├─────────────────────────────┤
│ [想要]  [申请接漂]            │
└─────────────────────────────┘
```

#### 删除 / 调整

| 现状 | 调整后 |
|------|--------|
| 底部 `tip` 长段（到付 + 线下结算 + 平台不收费 + 实际运费） | **删除**，核心 2 句已进 know-card；细节进「了解更多」 |
| know-card 3 条独立短句 | **合并为 3 条**，第 1 条合并占用+记录 |
| 无 5 单说明 | **新增**第 3 条（用户在详情页即可判断能否接） |
| `detail-ship-line`「快递到付」 | 保留，与 know 不冲突（一行 meta vs 规则卡片） |

#### 目标文案（canClaim 时）

**接漂须知**

1. 先占用 **{{coinValue}}** 公益积分，确认收货后完成积分记录  
2. 快递到付，平台不代收运费  
3. 在途接漂最多 5 单  

**了解更多**：`查看漂流玩法 ›` → `/pages/drift/guide`（与漂流池首页「玩法介绍 ›」同一页，见 §10）

### 3.3 确认弹窗（`buildClaimConfirmContent`）

#### 现状 vs 目标

**现状（3 行）**

```
占用 12 公益积分
完成接漂收货后，将从总记录值中扣除对应公益积分
是否确认申请？
```

**目标（3 行，更短、与页面术语一致）** — ✅ 已确认

```
先占用 12 公益积分
确认收货后从占用中结算

是否确认申请？
```

| 调整 | 原因 |
|------|------|
| 「占用」→「先占用」 | 与 know-card / 摘要动词一致 |
| 「从总记录值中扣除」→「从占用中结算」 | 去掉「扣除」语义，与占用动作配对 |
| 空一行再「是否确认」 | 动作与规则视觉分组 |

**弹窗不出现**：到付、5 单、合并寄出、平台收费 — 均已在前两步披露。

**成功 toast**：保持 `申请已提交`（contract 要求简洁）。

## 4. 信息出现次数对比

| 信息 | 优化前（全流程） | 优化后 |
|------|----------------|--------|
| 占用公益积分 | 详情 2 + 申请 3 + 弹窗 1 ≈ **6** | 详情 1 + 申请 1 + 弹窗 1 = **3** |
| 收货后记录/扣除 | 详情 1 + 申请 2 + 弹窗 1 ≈ **4** | 详情 1 + 弹窗 1 = **2**（摘要不重复讲记录） |
| 快递到付 | 详情 3 + 申请 4 ≈ **7** | 详情 2 + 申请 1 = **3** |
| 平台不收费 | 详情 1 + 申请 1 ≈ **2** | 详情 1 = **1** |
| 5 单上限 | 申请 2 = **2** | 详情 1 + 申请 1 = **2**（位置更合理） |
| 同城/跨省 hint | 申请 1～2 | 申请 0～1（仅动态区） |

## 10. 「查看漂流玩法 ›」跳转说明

### 推荐跳转：`/pages/drift/guide`

与漂流池首页已有入口一致：

```javascript
// miniprogram/pages/pool/index.js
goGuide() {
  wx.navigateTo({ url: '/pages/drift/guide' });
}
```

首页文案为「玩法介绍 ›」，详情页用「查看漂流玩法 ›」，**同一目的地、略偏接漂视角**，避免新增页面。

| 维度 | `pages/drift/guide` 现状 |
|------|--------------------------|
| 导航栏标题 | 漂流玩法 |
| 内容侧重 | 情感价值、赠书三步流、72h 发货 / 10 天收货、发货前可取消、公益积分结算 |
| 已有 CTA | 「去看看正在漂流的书」→ 回流漂流池 tab |
| **未覆盖** | 快递到付、平台不代收运费、在途 5 单上限、占用机制、合并寄出 |

### 与详情「接漂须知」的分工

| 层级 | 位置 | 职责 |
|------|------|------|
| 决策必知 | 详情 know-card 3 条 | 占用、到付、5 单 — **接漂前必须看到** |
| 延伸阅读 | guide 页 | 玩法故事、时效约定、积分结算总则 — **有兴趣再点** |

因此：**删掉详情底部 long tip 是安全的**，关键规则已在 know-card；guide 承担「为什么漂 / 整体怎么玩」，不必重复 know-card 三条。

### 实现要点（后续开发）

```javascript
// pool/detail.js
goGuide() {
  wx.navigateTo({ url: '/pages/drift/guide' });
}
```

```xml
<!-- pool/detail.wxml，接漂须知卡片下方 -->
<view wx:if="{{item.canClaim}}" class="guide-link" bindtap="goGuide">查看漂流玩法 ›</view>
```

样式对齐首页 `.guide-entry`（品牌绿、文字链、不换行）。

### 可选增强（非 v1 必须）

若希望 guide 与详情删掉的 tip 完全衔接，可在 guide「几条简单约定」中**追加 2 条**（实现文案优化时再改 guide.wxml）：

- 运费为快递到付，平台不代收运费，实际运费以快递员收费为准  
- 在途接漂最多 5 单，请先完成未收货记录  

**v1 可不改动 guide 页**，仅加链接；接漂关键信息已由 know-card 覆盖。

## 9. 验收标准

- 接漂全流程用户可见的规则性段落 ≤ 6 处（know 3 条 + 摘要 6 行 + 0～1 hint + 弹窗 3 行）
- 三页 + 弹窗共用 `driftCopy.js`，无硬编码重复长句
- 同赠书方区块单组件维护
- 本地 contract 全绿
