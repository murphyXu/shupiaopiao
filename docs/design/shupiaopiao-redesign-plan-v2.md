# 书漂漂小程序设计与交互改造方案（修订版 v2）

更新时间：2026-06-28
修订说明：本版基于对现有代码（`pool/`、`shelf/`、`mine/` 等页面）的核对，修正了四处主要问题——①统一漂流广场口径并明确这是筛选结构替换；②全局统一“公益积分”字段标签，消除“流转积分”混用；③书架筛选选型定稿；④新增边界/异常态交互规范。其余内容沿用前序方案。

终稿微调（2026-06-28 第二次确认）：
1. 漂流广场价值主张文案定为“闲置漂出去，好书接回来”。
2. 书架页本版暂不支持书架名称编辑，移除“编辑名称”入口。
3. “我的”页本版保持现状：顶部统计区不变；待办继续沿用现有“我的漂流（赠出）/（领取）”菜单内联 badge 方式，不新增“待处理”聚合模块（§9 相关建议降级为后续可选）。

## 1. 改造目标

本方案用于指导后续 UI、交互和轻量业务逻辑调整。当前阶段仅作为设计方案文档，不代表已执行代码修改。

核心目标：

- 让新用户在首屏快速理解：闲置书可以上漂，接漂需要公益积分，运费采用快递到付。
- 让老用户更快完成两条主路径：接漂找书、从书架上漂赠书。
- 保持现有合规边界：不使用购买、余额、交易、提现等价格化表达；不新增读书笔记、公开社交流、用户自定义图片上传等敏感能力。
- 保持现有产品术语：公益积分、信用积分、占用中、快递到付。

### 1.1 术语统一约定（本版新增）

为避免歧义，全文统一如下：

- **公益积分**：唯一对用户展示的积分字段标签。书卡、详情、申请摘要、明细中标注积分数值时，一律使用“公益积分”，不再出现“流转积分”作为字段标签。
- **“流转”仅作语义描述**：用于解释积分“站内流转、不具现金属性”的语境（如“接漂时先占用，完成后记录结算”），不作为任何数值的标签词。
- **信用积分**：履约相关，独立于公益积分，标签固定为“信用积分”。

## 2. 本版修正原则

相对于前序方案，本版固定以下要点：

1. 漂流页的一句价值主张和“玩法介绍”入口与“漂流广场”标题并排一行，通过字号和按钮样式区分。
2. 漂流广场的可用积分和“获得积分”入口与当前版本保持一致，不重构成新的积分卡片。
3. 漂流广场的筛选为**结构性替换**：由当前的“按品类 / 按价值 / 按品相”替换为“推荐 / 具体分类 / 更多筛选”（详见 §4）。
4. 书架页面筛选选型定稿为“推荐 / 书籍分类（折叠）/ 更多筛选”（详见 §8）。
5. 新增 §15 边界与异常态交互规范。

除以上要点外，其他页面建议沿用前序评估方案。

## 3. 全局设计原则

### 3.1 信息层级

- 每个页面最多突出一个主行动按钮。
- 主按钮只用于推进核心流程：申请接漂、确认申请、提交上漂、去发货。
- 次按钮用于低承诺行为：想要、查看规则、稍后处理、查看详情。
- 绿色只用于主行动和公益积分相关信息，不用于所有标签。
- 警示信息使用浅黄或浅橙，避免和公益积分的薄荷绿混用。

### 3.2 文案原则

- 避免交易化表达：不使用“购买”“价格”“付款给平台”“余额”等词。
- 运费说明保持明确但克制：可说明“快递到付”“收件时向快递员支付”，不展示具体金额估算。
- 公益积分说明保持站内流转语义：接漂先占用，完成后记录结算。
- 信用积分说明保持履约语义：发货、收货、申诉、取消等行为会影响履约记录。

### 3.3 导航与任务优先级

- 首页优先承接“找书接漂”。
- 书架优先承接“管理藏书”和“从已有书发起上漂”。
- 我的页优先承接待办，包括待发货、待确认、待评价、申诉等。

## 4. 漂流广场

对应页面：`miniprogram/pages/pool/index`

### 4.1 当前状态（已核对代码）

- 当前一级筛选为 `按品类 / 按价值 / 按品相`（代码 `FILTER_MODES`），二级随之切换为品类 / 价值区间 / 品相。新用户需要先理解三个维度才能筛书，认知成本偏高。
- 头部“漂流广场 / 玩法介绍 / 可用积分 / 获得积分”结构已稳定，本版保留。
- 书卡当前已渲染“{{coinValue}} 公益积分 + 分类”，主行动为“申请接漂”，均合理，保留。

> 说明：本版对筛选是**结构性替换**而非微调，需同步调整 `FILTER_MODES`、`secondaryTabsFor` 及对应筛选逻辑，详见 §12.1。

### 4.2 改造方案

首屏结构建议：

```text
漂流广场   闲置漂出去，好书接回来   [玩法介绍]
剩余可用公益积分 12                 [获得积分]

搜索书名 · 作者 · ISBN

推荐 / 童书 / 文学 / 社科 / 更多筛选

双列图书卡片
书封
书名
公益积分 + 分类
[申请接漂]

悬浮入口：[＋ 上漂赠书]
```

### 4.3 重点修正

- “漂流广场”、一句价值主张、“玩法介绍”在同一行；“漂流广场”继续作为大标题。
- 价值主张使用小于主标题的字号、中性灰绿色、单行省略（具体字号由视觉规范统一定义，不在本文钉死像素值）。
- “玩法介绍”继续保持按钮/胶囊样式，与文字主张形成视觉区分。
- 可用公益积分和“获得积分”入口保持当前结构：一行展示，不改为积分卡片。
- 书卡标值统一使用“公益积分”，不出现“流转积分”字样。

### 4.4 筛选建议

- 默认露出高频入口：推荐、童书、文学、社科、更多筛选。
- “更多筛选”承载原“按价值”的价值区间和“按品相”筛选，确保能力不丢失。
- 列表主按钮继续使用“申请接漂”，不在列表卡片上使用“想要接漂”作为主入口。

### 4.5 空状态文案

```text
漂流池暂时没有符合条件的书
```

按钮：

```text
去上漂一本书
```

## 5. 漂流详情

对应页面：`miniprogram/pages/pool/detail`

### 5.1 当前问题

- “想要接漂”和“申请接漂”并列时，用户可能误判两者是同等优先级。
- 积分占用、运费到付、寄出地、品相、赠书方信用等信息分散，需要用户自行理解规则。
- “举报”入口需要保留，但不应抢占核心决策区域。

### 5.2 改造方案

页面结构建议：


## 7. 上漂赠书

对应页面：`miniprogram/pages/drift/publish`

### 7.1 当前问题

- 当前表单较长，用户不容易判断完成上漂需要几步。
- 批量上漂能力有价值，但首次上漂时容易提高理解成本。
- 定价缺失、补录信息、系统建议积分之间的关系需要更清晰。

### 7.2 改造方案

页面结构建议（步骤条按“是否多选”自适应）：

```text
步骤：选书 → 品相 → 积分 → 确认

1 选择要送出的书
书籍列表 / 已选状态 / 可继续选择多本

2 描述品相
几乎全新 / 良好 / 明显使用痕迹
可选：有笔记 / 有折角 / 有脏污 / 有破损

3 公益积分
系统按图书定价和品相给出建议积分
本次设定：[-] 6 [+]

4 漂流设置
匿名漂流 / 昵称漂流
寄出参考地

[提交上漂]
```

### 7.3 交互建议

- 单本上漂为默认主路径；步骤条始终展示四步，但单本场景下“选书”步骤在选定后即折叠为摘要，避免与“强调单本”冲突。
- 多选保留为增强能力，仅在用户主动继续添加时展开多本管理。
- 定价缺失应在选择阶段明确阻断或强提示，不建议提交时静默自动取消。
- 仅在用户将公益积分设为 0 时增加确认弹窗，避免普通流程过重。
- 匿名漂流继续作为默认选项，减少用户心理负担。

### 7.4 关键文案

页面标题：

```text
上漂赠书
```

选书标题：

```text

```


## 8. 我的书架

对应页面：`miniprogram/pages/shelf/index`

### 8.1 当前状态（已核对代码）

- 当前一级筛选为 `书籍分类 / 阅读状态 / 书架位置`（代码 `PRIMARY_TABS`），二级随之切换。默认对普通用户偏“管理维度”，不够直觉。
- 添加入口、容量兑换、系列折叠等能力已具备，保留。
- 书架与“从已有书发起上漂”的连接还可以更强。
- 本版暂不支持书架名称编辑，移除标题行的“编辑名称”入口（数据字段保留，后续可恢复）。

### 8.2 筛选选型定稿（本版拍板）

本版定稿采用**“推荐 / 书籍分类（折叠）/ 更多筛选”**，理由：避免一级 tab 过长在窄屏溢出，改动相对可控，同时保留全部分类能力。

一级筛选：

```text
推荐 / 书籍分类 / 更多筛选
```

二级筛选：

```text
- 推荐：全部
- 书籍分类：全部 / 童书 / 文学 / 社科 / 经管 / 科普 / 生活 / 艺术 / 其他
- 更多筛选：阅读状态 + 书架位置
```

其中：

- “推荐”默认展示全部或按后续推荐规则排序后的藏书。
- “书籍分类”作为一级入口直达二级品类，减少“先选维度再选分类”的层级。
- “更多筛选”承载原有阅读状态和书架位置能力，确保不丢失。
- “可上漂 / 已上漂”作为后续增强（P2），落地时基于 `canPublishDrift` 和漂流状态计算。

### 8.3 书架卡片建议

书卡结构建议：

```text
书封
书名
作者 · 阅读状态
[上漂这本] / [查看漂流]
```

### 8.4 交互建议

- 书卡轻触仍进入书籍详情。
- 卡片操作区可直达“上漂这本”，并携带 `shelfBookId` 预选对应书籍（见 §12.3）。
- 已上漂书籍的操作切换为“查看漂流”。
- 如果容量不足，添加入口先提示“容量不足，可用公益积分兑换额度”。

### 8.5 空状态文案

无可上漂书：

```text
这里还没有可上漂的书
```

按钮：

```text
去添加书籍
```

## 11. 公益积分明细

对应页面：`miniprogram/pages/mine/wallet`

### 11.1 当前问题

- “总记录值 / 占用中 / 可用公益积分”的关系对普通用户不够直观。
- 明细列表直接展示流水，对规则解释帮助有限。

### 11.2 改造方案

页面结构建议：

```text
可用公益积分
12
占用中 6 · 累计记录 38
[获得积分]

可用 = 累计记录 − 占用中 − 已扣除

公益积分怎么用
- 接漂时先占用
- 收货完成后记录扣除
- 上漂完成后记录增加

明细
完成赠书记录 +6
接漂占用
取消接漂释放占用
```

### 11.3 文案建议

- “总记录值”改为“累计记录”。
- 顶部补一行关系说明：`可用 = 累计记录 − 占用中 − 已扣除`，消除三值困惑。
- “余额”类文案继续避免使用。
- 明细描述统一为公益积分语义。

建议流水类型文案：

```text
完成赠书记录
接漂占用
取消接漂释放占用
确认收货记录扣除
申诉关闭释放占用
```

### 11.4 后续增强

可增加明细筛选：

```text
全部 / 增加 / 占用 / 释放 / 扣除
```

该项可作为 P2，不影响首批改造。

## 12. 非 UI / 交互层面的逻辑改动

### 12.1 漂流广场

- 筛选由“按品类/按价值/按品相”替换为“推荐/具体分类/更多筛选”，需重写 `FILTER_MODES`、`secondaryTabsFor` 及 `filterList` 中的价值/品相过滤分支，使其归入“更多筛选”。
- 新增“推荐”筛选需要定义推荐排序策略，V2 首版仅取主因子：**可接漂优先 + 有封面优先**；其余因子（轻量优先、近距离优先、赠书方信用优先）作为 P2。
- “更多筛选”需要承载原有价值区间和品相筛选，避免能力丢失。

### 12.2 申请接漂

- 申请页需要聚合地址、可用公益积分、占用积分、寄出地、未收货上限。
- 确认按钮状态应依赖：是否选择地址、可用公益积分是否足够、是否达到未收货上限。
- 公益积分不足时，应提前引导“获得积分”或“加入想要”，减少提交后失败。
- 提交接口需返回明确的失败码（已被接走 / 积分不足 / 超未收货上限），前端按 §15 分别提示。

### 12.3 上漂发布

- 发布页需要维护步骤状态：选书、品相、积分、确认。
- 定价缺失应前置阻断或明确提示，不建议提交时静默自动取消。
- 从书架卡片进入上漂时，需要携带 `shelfBookId` 并预选对应书籍。

### 12.4 书架

- 采用“推荐 / 书籍分类 / 更多筛选”，需调整 `PRIMARY_TABS` 与 `refreshSecondaryTabs`：一级减为三项，原“阅读状态/书架位置”并入“更多筛选”。
- “更多筛选”需要保留原有阅读状态和书架位置能力。
- 如新增“可上漂 / 已上漂”筛选（P2），需基于 `canPublishDrift` 和漂流状态计算。

### 12.5 我的与订单

- 我的页待办模块需要按状态聚合，并支持点击直达订单列表对应 tab。
- 直达路由约定：`/pages/drift/given?tab=待发货`、`/pages/drift/received?tab=待确认` 等，参数为状态 key。
- 订单卡需要增加或前端计算 `nextStep` 文案。
- 合并发货卡需要展示 `orderCount`、同一收件地址、合并发货入口。

### 12.6 公益积分

- “累计记录”仍可沿用现有 `balance` 数据，只调整展示文案。
- 顶部关系说明 `可用 = 累计记录 − 占用中 − 已扣除` 为纯前端展示，不改数据结构。
- 明细筛选如落地，需要按交易类型归类：增加、占用、释放、扣除。
- 所有积分流水描述继续避免钱包、余额、购买式表达。

## 14. 验收标准

### 14.1 体验验收

- 新用户进入漂流广场后，能在首屏看到“漂流广场”、价值主张、玩法介绍、可用积分和获得积分入口。
- 用户在漂流详情页能明确知道：申请会占用公益积分，运费为快递到付，完成后再结算积分记录。
- 用户在申请页确认前能看到本次申请摘要。
- 用户在书架页能更快按推荐或具体分类浏览藏书，并能在更多筛选中找到阅读状态和书架位置。
- 用户在我的页能优先看到待处理事项。

### 14.2 合规验收

- 不出现“购买”“提现”“余额”“付款给平台”等交易化表达。
- 不出现“流转积分”作为字段标签（统一为“公益积分”）。
- 不新增公开笔记、公开评论流、用户图片社交分享能力。
- 运费说明不展示具体金额估算。
- 公益积分仍定义为站内流转记录，不具备现金属性。

### 14.3 技术验收

- 页面 tab 路由结构不变；订单列表等页面允许新增查询参数（如 `tab`）。
- 现有接漂、上漂、发货、确认收货流程不被破坏。
- 原有合同测试需按新产品约束更新：尤其是漂流首页头部布局、漂流广场筛选结构、书架筛选默认项。
- 若涉及云函数数据字段变化，应补充对应合同测试；仅前端展示调整则优先保持接口不变。
- §15 失败态需有对应接口失败码约定与前端兜底测试。

## 15. 边界与异常态交互规范（本版新增）

正向主流程之外，以下异常态需有明确反馈，避免用户停留在加载态或遇到无解释的失败。

### 15.1 申请接漂失败

| 场景 | 触发 | 提示文案 | 后续动作 |
|---|---|---|---|
| 书已被接走 | 提交瞬间该漂流已被他人接走 | 这本书刚刚被接走了，去看看其他书 | 返回漂流广场并刷新列表 |
| 积分被并发占用 | 提交时可用公益积分不足 | 可用公益积分不足，去获得积分 | 引导“获得积分”或“加入想要” |
| 超未收货上限 | 达到最多未收货单数 | 未收货已达上限，先完成收货再申请 | 引导“我接到的书” |
| 未选地址 | 无收货地址 | 先选择收货地址 | 聚焦地址卡 |
| 网络失败 | 请求超时/断网 | 网络不太稳定，请重试 | 保留页面状态，提供“重试” |

### 15.2 地址读取失败

- 微信地址授权失败或读取异常时，提示“暂未获取到微信地址，可手动填写”，不阻断流程。

### 15.3 合并发货中途变更

- 合并包裹中某单被取消时：实时更新“合并 N 本”数量；若仅剩 1 单，去除合并标识，恢复单单展示；不整卡报错。

### 15.4 上漂提交异常

- 定价缺失：选择阶段即阻断并提示“缺少定价，补录后才能上漂”，不在提交时静默取消。
- 提交失败（网络/服务）：保留已填步骤状态，提供“重试”，不清空表单。

### 15.5 通用兜底

- 所有主按钮在请求进行中显示 loading，禁止重复提交。
- 任何失败均给出可读文案 + 明确下一步，禁止仅 toast “失败”而无指引。
